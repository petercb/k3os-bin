# Replace manual cmdline parsing with u-root CmdlineParser interface

The codebase previously scattered kernel command-line parsing across four packages, each doing its own `strings.Fields` + prefix matching against `/proc/cmdline`. This change introduces a `CmdlineParser` interface in `internal/iface`, implements it via u-root's `pkg/cmdline` in `internal/cmdline`, and migrates all four consumers to inject the interface. The result is a single parse, no repeated file reads, and fully mockable cmdline access in tests.

**Watch for:** The `enterchroot` package uses a package-level `cmdlineParser` var instead of constructor injection (confirmed). `Consoles()` is exposed in the interface but unused in production code (confirmed). Mock struct duplicated four times across test packages (confirmed, minor).

## High-level view

The single `cmdline.New()` call in `main.go` creates the parser once and threads it to all consumers, eliminating repeated `/proc/cmdline` reads during boot. The interface exposes `Flag`, `Contains`, `Consoles`, and `Raw` without leaking u-root types.

Mode detection (`detect.go`) benefits most: the old `parseCmdline()` + `CmdlineResult` struct is replaced by direct `Flag`/`Contains` calls. Error handling shifted ‚Äî the old `CmdlineReader` propagated read errors as hard failures, while the new interface has no error returns. If u-root fails to read `/proc/cmdline`, it returns empty state, and mode detection falls through to the retry loop instead of failing immediately.

The `enterchroot` package deviates from the constructor-injection pattern used elsewhere. It declares `cmdlineParser` as a package-level var initialized at import time, which reads `/proc/cmdline` eagerly. The dependency is invisible to callers of `Mount()`, and tests must mutate the package-level var (preventing `t.Parallel()`).

The TTY setup in `finalize/ttys.go` uses `Raw()` and still manually parses `console=` entries to extract baudrate. `Consoles()` exists in the interface but returns unsplit values like `"ttyS0,115200n8"`, so `parseConsoleEntries` is still needed. No production code currently calls `Consoles()`.

## Package-level parser initialization in enterchroot

The `enterchroot` package initializes `cmdlineParser` at the `var` block level:

```go
var cmdlineParser iface.CmdlineParser = cmdline.New()
```

This calls `uroot.NewCmdLine()` which reads `/proc/cmdline` at package init time. The other three consumers (`boot.Init`, `finalize.Finalizer`, `mode.Detector`) accept `CmdlineParser` as a struct field injected by the caller. The enterchroot deviation means:

- The dependency is invisible to the constructor ‚Äî you can't tell from `Mount()`'s signature that it reads `/proc/cmdline`.
- Tests override it by mutating the package-level var and restoring via `t.Cleanup`, which prevents `t.Parallel()` on those tests (though the tests don't currently declare parallel).

The pattern matches pre-existing `loopAttacher` and `ensureLoopFn` vars in the same file, so it's a deliberate consistency within the package even if it diverges from the rest of the codebase.

## Error-path change in mode detection

The old `Detector` propagated cmdline read errors as immediate failures:

```go
// old: hard failure
cmdline, err := d.CmdlineReader()
if err != nil { return "", err }
```

The new interface has no error returns. If u-root fails to read `/proc/cmdline`, it returns empty values ‚Äî mode detection silently falls through to the retry loop and eventually times out. `TestDetect_EmptyParserFallsThrough` validates this: an empty parser with no other signals triggers the timeout path. This is a behavioral change from immediate-fail to timeout-then-fail, which is more resilient in early boot where `/proc` might not be ready on the first attempt, but slower to surface misconfiguration.

## `parseToMap` reimplementation

`NewFromString` populates the `AsMap` field of `uroot.CmdLine` by replicating u-root's unexported `parseToMap` logic (quoted-field handling, dash/underscore canonicalization). This creates coupling to u-root's internal parsing rules ‚Äî if u-root changes its map-building logic in a future version, the `NewFromString` test path could diverge from the `New()` production path. Since `NewFromString` is documented as test-only, the blast radius is limited to test fidelity.

## Test coverage

**Not tested:** `Consoles()` is never exercised through a production code path. Integration between `parseConsoleEntries` (which splits `Raw()` by hand) and `Consoles()` (which returns TTY names from u-root) is not tested ‚Äî if they disagree on what constitutes a console device name, nothing catches it.
