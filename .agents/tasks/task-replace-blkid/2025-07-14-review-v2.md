# Replace blkid/lsblk shell-outs with BlockProber interface (v2)

This change removes all `blkid` and `lsblk` process invocations from the boot sequence, replacing them with a `BlockProber` interface backed by direct sysfs/devfs reads. A follow-up refactoring commit addressed the four issues from the v1 review: the virtual device filter now excludes `zram`, `nbd`, `sr`, `md`; dead helper functions are removed; the detector closure references the already-wired instance; and `FindByLabel` rejects labels containing `/` or `..`.

Watch for: the `sr` prefix filter will also exclude any hypothetical device whose name starts with "sr" (e.g., a SCSI disk named `srd0` — not a real Linux naming convention, but the filter is broader than necessary) (possible). No functional regressions found.

**Verdict**: APPROVED

## High-level view

The sysfs/devfs approach replaces fork/exec of `blkid` and `lsblk` with atomic symlink resolution and directory listing. The interface is two methods — `FindByLabel` and `ListDisks` — covering the two block-discovery operations the boot code needs. All four v1 issues are resolved in the refactoring commit.

The virtual device filter in `ListDisks` now matches the old `lsblk --type disk` semantic closely enough for the boot-time use case. The only remaining concern is that `strings.HasPrefix(name, "sr")` is a broad prefix match — CD-ROM devices are always named `sr[0-9]+` in Linux, so no legitimate physical disk would be caught, but the pattern is worth noting for future maintainers.

## V1 issue resolution

All four issues from the v1 review are addressed in commit `0c12a33`: expanded `ListDisks` filter, removed dead helpers and their tests, simplified detector closure to a method value, and added label sanitization with test coverage.

## ListDisks `sr` prefix breadth

SCSI naming in Linux uses `sd` for disks and `sr` exclusively for CD-ROM. The prefix match `strings.HasPrefix(name, "sr")` is broader than "sr followed by digits" — a hypothetical future device named `srd0` would be incorrectly filtered. No such device exists in any shipping Linux kernel, making this cosmetic. If precision is desired, a regex or digit check after the two-character prefix would tighten it.
