# Replace blkid/lsblk shell-outs with BlockProber interface

This change removes all `blkid` and `lsblk` process invocations from the boot sequence, replacing them with a `BlockProber` interface backed by direct sysfs/devfs reads. The interface exposes `FindByLabel` (reads `/dev/disk/by-label/<label>` symlinks) and `ListDisks` (reads `/sys/block/` with virtual device filtering). The implementation is wired through the established `Deps` / `Finalizer` struct pattern, and the old `blockProbe` helper in main.go is deleted.

Watch for: incomplete virtual device filter in `ListDisks` may surface `zram`/`nbd`/`sr` devices that `lsblk` would have excluded (confirmed). Dead code (`parseDisks`, `splitLines`, `splitFields`) remains in live.go (confirmed).

## High-level view

The sysfs/devfs approach eliminates the fork/exec overhead and util-linux dependency — symlink resolution for labels is atomic and the right primitive for an init binary. The interface is minimal: two methods covering the two operations the boot code actually needs.

`ListDisks` filters `loop`, `ram`, and `dm-` prefixes from `/sys/block/`, but the old `lsblk -o NAME,TYPE -n` path filtered by the "disk" type column — which also excludes `zram`, `nbd`, `sr` (CD-ROM), and `md` (RAID) entries that appear in `/sys/block/`. The new filter is looser than the old behavior.

The `mode.Detector` wraps the prober in a closure to match its `func(label string) (string, error)` field type. The closure allocates a fresh `SysfsBlockProber{}` per call rather than closing over the instance already wired into `modeDeps` — functionally harmless (zero-size struct) but inconsistent with the other injection sites.

Three helper functions (`parseDisks`, `splitLines`, `splitFields`) are now dead code in live.go — still tested but no longer called from production paths.

## ListDisks filter gap versus lsblk behavior

The old code called `lsblk -o NAME,TYPE -n` and then filtered lines where the TYPE column equals `"disk"`. This excluded partitions, CD-ROM devices (`sr0`), RAID arrays (`md0`), network block devices (`nbd0`), compressed RAM devices (`zram0`), and device-mapper entries — only physical disks appeared.

The new implementation reads `/sys/block/` and filters three prefixes: `loop`, `ram`, `dm-`. But `/sys/block/` also exposes:

```
sr0       (CD-ROM drives)
zram0     (compressed RAM swap)
nbd0-15   (network block device)
md0       (software RAID)
```

On a system with an optical drive or zram configured, `ListDisks` will return these names. The live-boot retry loop will then attempt to mount `/dev/sr0` or `/dev/zram0`, which will fail but wastes retry iterations and could produce confusing log noise. More critically, if a CD-ROM auto-mounts successfully, the boot could bind to the wrong medium.

To match the semantic guarantee that callers expect (physical disks only), the filter should either:
- Add `sr`, `zram`, `nbd`, `md` to the prefix list, or
- Read `/sys/block/<name>/queue/rotational` or `/sys/block/<name>/device/type` as a positive "this is a real disk" signal.

The prefix approach is simplest and matches the scope of this change.

## Dead code: parseDisks and friends

`parseDisks`, `splitLines`, and `splitFields` in `live.go` are no longer called from any production path — `SetupBase` now receives pre-parsed disk names from `BlockProber.ListDisks()`. The FEAT-001 findings note they were left "as they are still tested directly," but testing dead code doesn't justify keeping it. Removing the functions and their tests reduces maintenance surface.

## Detector closure allocates unnecessarily

In `main.go`, the `ModeDetector` closure builds the `mode.Detector` with:

```go
BlockProber: func(label string) (string, error) {
    bp := osimpl.SysfsBlockProber{}
    return bp.FindByLabel(label)
},
```

`modeDeps` already holds an instance (`BlockProber: osimpl.SysfsBlockProber{}`). Closing over that — `modeDeps.BlockProber.FindByLabel` — would make it obvious to a reader that the detector uses the same prober as the mode handlers.

## Label path construction

`FindByLabel` joins the label into a path without validation:

```go
linkPath := filepath.Join("/dev/disk/by-label", label)
```

All current callers pass compile-time string literals (`"K3OS"`, `"K3OS_STATE"`), so there's no runtime risk today. A label like `"../by-uuid/something"` would resolve to an unintended symlink target. If this interface is ever exposed to user-controlled input (config files, cmdline parameters), a simple `strings.Contains(label, "/")` guard would close the door. Low-priority hardening, not blocking.
