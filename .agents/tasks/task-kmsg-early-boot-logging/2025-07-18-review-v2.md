# kmsg-backed early boot logging via internal/klog (v2)

Early boot messages were lost to a disconnected stderr when running as PID 1 on bare initramfs. This change introduces `internal/klog`, a thin wrapper around `/dev/kmsg` that routes all `slog` output to the kernel ring buffer so messages surface in `dmesg`. The package uses `go-kmsg` v0.1.6 for line splitting/truncation, falls back to `os.Stderr` transparently when `/dev/kmsg` is unavailable, and plumbs a shared `slog.LevelVar` through the enterchroot and init phases so level control survives the exec boundary.

Watch for: Non-debug `initrd()` path still has no kmsg logging during `Mount()` — only the debug guard enables it (confirmed, intentional per design doc but reduces observability for production boot failures).

**Verdict**: APPROVED

## High-level view

The `internal/klog` package opens `/dev/kmsg` with `O_WRONLY` (fixed from v1), wraps it in a `kmsg.Writer`, installs a `slog.TextHandler`, and exposes level control via a shared `LevelVar`. The fd is never explicitly closed in production — `O_CLOEXEC` ensures cleanup at the next `syscall.Exec`.

Three call sites invoke `Setup()` independently: `Enter()` (pre-pivot, unconditional), `Mount()` (debug-only), and `postChroot()` (post-exec). Each runs in a separate process image due to exec boundaries. The remaining observability gap is that `Mount()` — which resolves squashfs, attaches loop devices, and calls `PivotRoot` — only gets kmsg visibility when debug is enabled.

## Mount() observability gap in non-debug path

`Mount()` gates `klog.Setup()` behind `isDebug()`. In production (non-debug), `slog` calls inside `Mount()` hit Go's default handler targeting stderr — the same "lost to nowhere" condition that motivated this change. The `Enter()` path calls `Setup()` unconditionally, but it runs only after exec. The `Mount()` phase that resolves squashfs images, attaches loop devices, and calls `PivotRoot` has no kmsg visibility unless debug is enabled.

As a follow-up, moving `klog.Setup()` to the top of `initrd()` (before `Mount()`) would close this gap without changing the current change's contract.

All four v1 concerns addressed: `O_WRONLY` applied, `Mount()` assumption documented with comment, integration tests accepted as impractical (94% unit coverage sufficient), `siderolabs/gen` v0.5→v0.8 bump validated by passing builds.
