# kmsg-backed early boot logging via internal/klog

Early boot messages were lost to a disconnected stderr when running as PID 1 on bare initramfs. This change introduces `internal/klog`, a thin wrapper around `/dev/kmsg` that routes all `slog` output to the kernel ring buffer so messages surface in `dmesg`. The package uses `go-kmsg` v0.1.6 for line splitting/truncation, falls back to `os.Stderr` transparently when `/dev/kmsg` is unavailable, and plumbs a shared `slog.LevelVar` through the enterchroot and init phases so level control survives the exec boundary.

Watch for: `O_RDWR` is used where `O_WRONLY` would suffice, granting unnecessary read capability on the kernel ring buffer (confirmed); `Mount()` conditionally calls `klog.Setup()` inside a debug guard without closing a prior handle if one existed (confirmed, currently benign); the `siderolabs/gen` transitive dependency jumped from v0.5.0 to v0.8.6 which could break `go-blockdevice/v2` (likely).

## High-level view

The `internal/klog` package opens `/dev/kmsg`, wraps it in a `kmsg.Writer`, installs a `slog.TextHandler`, and exposes level control via a shared `LevelVar`. It is designed as a fire-and-forget bootstrap step ‚Äî `Close()` is exported for tests but never called in production because `O_CLOEXEC` ensures the fd is cleaned up at the next `syscall.Exec`.

The integration touches three call sites: `Enter()` (pre-pivot), `Mount()` (debug-only, same process as `initrd()`), and `postChroot()` (post-exec). Each calls `Setup()` independently, which is correct because exec boundaries create new processes. `Mount()` calling `Setup()` only within an `isDebug()` guard means the non-debug path has no kmsg logging at all during the `initrd()` phase ‚Äî intentional but worth noting for future debugging.

The `Init` struct gains a `LogLevel *slog.LevelVar` field. When non-nil (set by `postChroot()`), `setupDebug()` adjusts the same `LevelVar` that controls the kmsg handler, so enabling `k3os.debug` globally changes all slog output ‚Äî correct, but the coupling is implicit.

<summary>Issues (4)</summary>

1. **O_RDWR grants unnecessary read access** (confirmed) ‚Äî `/dev/kmsg` is opened read-write but only the write path is used. Use `os.O_WRONLY` to follow least-privilege. Read access on `/dev/kmsg` allows consuming kernel log entries, which is irrelevant here.
2. **Double Setup in Mount() could leak fd** (possible) ‚Äî `Mount()` calls `klog.Setup()` inside `if isDebug()` without checking whether a prior `Setup()` ran. Currently `Mount()` is only entered from `initrd()` which has no prior setup, so no actual leak exists. Add a comment documenting this assumption, or have `Setup()` accept an optional prior `*EarlyLogger` to close.
3. **No integration tests for klog wiring** (confirmed) ‚Äî `enterchroot` and `init` test suites don't exercise the `klog.Setup()` call path; unit tests only cover the `klog` package in isolation.
4. **Transitive gen v0.5 to v0.8 bump** (likely) ‚Äî `go-kmsg` pulled `siderolabs/gen` from v0.5.0 to v0.8.6. This is a major-version-sized jump in a v0 module. If `go-blockdevice/v2` (also a dependency) is pinned to gen v0.5 APIs, build breakage or subtle behavior changes are possible. Verify compatibility.

## /dev/kmsg open flags

`defaultOpenKmsg()` uses `os.O_RDWR|unix.O_CLOEXEC|unix.O_NONBLOCK|unix.O_NOCTTY`. The `O_CLOEXEC` is critical: both `Enter()` and `postChroot()` terminate via `syscall.Exec`, so the fd never leaks across process boundaries. The `Close()` method is effectively dead code in production ‚Äî it exists solely for test hygiene.

`O_RDWR` grants read access to the kernel ring buffer. The `kmsg.Writer` struct uses only the `Write` method of its `KmsgWriter` field. Switching to `os.O_WRONLY` is a one-line fix that removes a capability the code doesn't need. In a PID 1 context this is defense-in-depth rather than a blocking issue.

## Mount() conditional Setup and global state

```go
// enter.go, Mount()
if isDebug() {
    logger := klog.Setup()
    logger.SetDebug()
    _ = os.Setenv("ENTER_DEBUG", "true")
}
```

`Mount()` is called from `initrd()`. At that point no prior `klog.Setup()` has run ‚Äî `Enter()` runs only after reexec. So in the current call graph there's no double-open. The fragility is structural: if someone later adds a `klog.Setup()` at the top of `initrd()`, `Mount()`'s conditional call would silently orphan the earlier fd (still `O_CLOEXEC`-protected at exec time, so no real leak, but confusing).

The `logger` variable is scoped inside the `if` block and immediately discarded. Only the slog global persists. This means the non-debug `initrd()` path has no kmsg logging at all ‚Äî messages during `Mount()` go to the prior slog default (Go's built-in handler targeting stderr), which is the same "lost to nowhere" problem that motivated this change.

## Test coverage gaps

The `klog` package unit tests use the `openKmsg` function-variable override pattern and achieve strong isolated coverage: success path, fallback, level control, close semantics, structured output, line splitting.

Not tested: the integration between `enter.go`/`main.go`/`init.go` and `klog.Setup()`. Those callers involve `unix.Setrlimit`, `syscall.Exec`, `PivotRoot` etc., making end-to-end testing difficult without namespace isolation. The `init.go` tests exercise only the `LogLevel: nil` fallback path.

The tests mutate the package-level `openKmsg` variable with `t.Cleanup` restoration. This is safe with sequential execution but would race if `t.Parallel()` were added to any of these tests.

## Dependency addition

`go-kmsg v0.1.6` is MPL-2.0 (file-level copyleft, not viral). It introduces `go.uber.org/goleak` as a transitive test dependency in `go.sum` and bumps `siderolabs/gen` from v0.5.0 to v0.8.6. That gen jump crosses what would be breaking-change territory in a post-v1 module. Since `go-blockdevice/v2` also depends on `siderolabs/gen`, the upgrade needs verification that the newer gen doesn't break blockdevice APIs used by this project.
