# Loopback init via netlink replacing /sbin/ip shell-outs

The `doLoopback()` function in the rc (run-control) package previously shelled out to `/sbin/ip` three times to configure the loopback interface during early boot. This change replaces those three `exec.Command` calls with a single call to `libinit.NetInit()` from the u-root project, which uses the netlink API to bring up `lo` directly. The old code also manually added 127.0.0.1/8 and a host route; the new approach relies on kernel 6.8+ auto-configuring the loopback address when the interface comes up.

Watch for: the kernel 6.8+ assumption is load-bearing — if this binary ever runs on an older kernel, loopback will come up without an address (confirmed). The AST-based tests verify source structure rather than runtime behavior, meaning a broken `NetInit` call would still pass tests (confirmed). Three new transitive compression dependencies (`klauspost/compress`, `klauspost/pgzip`, `ulikunitz/xz`) were pulled in by `go mod tidy` and have no relation to networking (confirmed).

## High-level view

The production change is one function body: seven lines of shell-out code become one call to `libinit.NetInit()`. The u-root implementation looks up `lo` by name via netlink and sets it up — no address assignment, no route addition. This is correct only when the kernel handles loopback address configuration automatically, which requires kernel 6.8 or later.

The test strategy uses Go's AST parser to inspect `rc.go` at test time, verifying structural properties of `doLoopback`. This cannot catch runtime failures but guards against regression to shell-outs, and the function requires root with real network namespaces to exercise properly.

The dependency footprint grew by five modules. `vishvananda/netlink` and `vishvananda/netns` are expected. The three compression libraries are unrelated — pulled in by `go mod tidy` resolving the full u-root module graph.

## Behavioral change: address configuration dropped

The old code performed: add 127.0.0.1/8 to `lo`, add host route for 127.0.0.0/8, then bring `lo` up. The new code performs only the last step via `netlink.LinkSetUp`. The address and route are expected to appear automatically via kernel behavior introduced in 6.8.

On kernels older than 6.8, loopback will come up but have no address assigned. Any process binding to 127.0.0.1 or connecting to localhost would fail. The plan document and status.md mention the kernel requirement, but the code itself carries no comment or guard. Since this is an init system for k3OS which presumably controls its kernel version, this may be acceptable — but the assumption should be documented at the call site for maintainability.

## Error handling: log stream divergence

The old code discarded all three `cmd.Run()` return values. The new code calls `libinit.NetInit()` which internally logs failures to `ulog.KernelLog` (the kernel ring buffer via `/dev/kmsg`). The rc package never sees the error and has no opportunity to log it alongside other rc failures that go to `log.Printf` (stderr).

If loopback init fails, the error message will appear in dmesg rather than in the same log stream as other rc failures. This makes debugging harder in environments where only application logs (not kernel logs) are captured.

## AST-based tests: coverage gap

The tests parse `rc.go` and verify two properties: (1) `doLoopback` calls `libinit.NetInit`, (2) it does not call `exec.Command`. A third test (`TestRcGo_ImportsLibinit`) checks that the import exists — this is redundant since the first test already implies correct compilation with the import, and the build itself would fail without it.

The tests cannot detect runtime failures: netlink unavailable, missing capabilities, absent `lo` interface. The plan document acknowledges this and relies on system-level integration tests. No behavioral test exists for the loopback initialization path.

## Transitive dependency growth

Five new entries in go.mod's indirect section:

- `vishvananda/netlink v1.3.0` and `vishvananda/netns v0.0.4` — expected transitive deps of libinit
- `klauspost/compress v1.18.0`, `klauspost/pgzip v1.2.6`, `ulikunitz/xz v0.5.15` — compression libraries with no relation to networking

The compression libraries are pulled in because `go mod tidy` resolves the entire u-root module graph. The practical concern is supply chain surface area: three additional indirect dependencies that this project will never import.
