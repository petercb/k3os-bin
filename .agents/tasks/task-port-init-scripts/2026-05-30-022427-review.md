# Shell-to-Go port of k3OS init sequence

The k3OS init pipeline — bootstrap, mode detection, mode-specific setup, boot finalization, exec /sbin/init — has been rewritten from ~10 shell scripts into a structured Go binary spanning 51 files and ~7100 lines. The design uses interface-driven DI (iface.FileSystem, iface.CommandRunner, iface.Mounter, modes.ProcessExecutor) to keep unit-testable seams at every OS boundary. Phases run sequentially via ordered step tables; any failure drops to a rescue bash shell, matching the original `set -e` + `trap rescue EXIT` semantics.

Watch for: missing `rc` step in the bootstrap phase (confirmed — the shell calls `k3os rc` but Go omits it); TTY deduplication skipped (confirmed — serial consoles can appear twice in inittab); console I/O redirect (`exec >/dev/console`) absent from the Go init (confirmed); `Finalizer.Mode` is set via a closure side-effect inside a lambda, which is invisible to callers.

## High-level view

The orchestrator (`internal/boot/init.go`) is a clean sequential pipeline with rescue-on-error, but the wiring in `main.go` hides a dependency mutation: the mode detector closure writes `fin.Mode` as a side-effect, coupling detection to finalization through a captured pointer rather than an explicit data path.

Bootstrap faithfully mirrors the shell except for one dropped step: the original calls `$K3OS_SYSTEM/k3os/current/k3os rc` between user setup and directory setup. This runs hardware initialization (modalias-based module loading, devtmpfs, mounts). Its absence means the Go init may boot without loading kernel modules that the `rc` step would have triggered.

The mode handlers (disk, local, live, install, shell) are the riskiest area. Disk mode's `Takeover()` can issue `reboot -f` or `poweroff -f` mid-sequence, then continues into `CleanupEphemeral()` and `PivotAndExec()` if those commands fail. The grow logic in both disk mode and the finalizer swallows errors from `parted`/`partprobe`/`e2fsck`/`resize2fs` without logging, making partition-grow failures invisible.

The boot finalization covers all steps from the shell script. The console I/O redirect (`exec >/dev/console`) that the shell performs after bootstrap is absent, so boot messages won't reach the physical console on a real initramfs boot.

<summary>Issues (7)</summary>

1. **Missing `k3os rc` in bootstrap** — The original bootstrap calls `$K3OS_SYSTEM/k3os/current/k3os rc` for hardware init (modalias module loading, devtmpfs). Add this step between SetupUsers and SetupDirs, or document why it's intentionally omitted.
2. **TTY serial console duplication** — The shell checks `grep -q "^${TTY}::" /etc/inittab` before adding serial entries. The Go code doesn't deduplicate, so if `ttyS0` is both a standard TTY device and in `console=ttyS0`, it appears twice in inittab. Add a seen-set check.
3. **Console I/O redirect missing** — The shell does `exec >/dev/console </dev/console 2>&1` after bootstrap. The Go port never redirects stdin/stdout/stderr to /dev/console, so early-boot output may go to the wrong fd.
4. **Finalizer.Mode mutation via closure** — In `main.go` postChroot, the mode detector closure sets `fin.Mode` as a side-effect. If detection errors partway through future refactoring, Finalizer runs with empty mode. Make mode an explicit return value wired through Init.
5. **Disk Takeover continues after reboot/poweroff** — `Takeover()` calls `reboot -f` at the end but is followed by `CleanupEphemeral()` and `PivotAndExec()` in `Execute()`. If reboot fails (dry-run, container), the sequence proceeds into pivot with potentially corrupted state. The shell's `cd $TARGET; takeover || true` pattern handles this differently.
6. **GrowPart errors swallowed silently** — In `DiskHandler.SetupMounts`, the grow commands (`parted`, `partprobe`, `e2fsck`, `resize2fs`) all discard errors via `_ =`. In the finalizer's `GrowLive`, errors from `parted`/`partprobe`/`resize2fs` are returned but `sleep 2` is implemented as a command (`f.Cmd.Run("sleep", "2")`). At minimum, log the grow errors.
7. **`sleep 2` via exec** — `GrowLive` calls `f.Cmd.Run("sleep", "2")` which spawns a subprocess. Use `time.Sleep(2 * time.Second)` or inject a sleep function like the mode detector does.

## Missing `k3os rc` hardware initialization

The original bootstrap script has this sequence after `setup_users`:

```
setup_users
$K3OS_SYSTEM/k3os/current/k3os rc
setup_dirs
setup_kernel
setup_config
```

The `rc` command (already implemented in `internal/cli/rc/rc.go`) sets up devtmpfs, loads kernel modules via modalias matching, mounts sysfs/procfs/cgroups, and configures basic devices. The Go `Bootstrapper.Run()` skips this entirely — the step table goes `SetupUsers -> SetupDirs -> SetupKernel -> SetupConfig`. On a real boot, this means module loading and device infrastructure established by `rc` won't happen until later (if at all). The kernel squashfs mount in `SetupKernel` may fail if devtmpfs isn't up, and network modules won't be loaded for DHCP-based config in initrd mode.

**Confidence: confirmed** — comparing `bootstrap.Run()` step list against the shell script line by line.

## TTY duplication in serial console setup

The shell script's `setup_ttys` checks before adding a serial console:

```bash
if [ -e /dev/${TTY} ] && ! grep -q "^${TTY}::" /etc/inittab; then
```

The Go `SetupTTYs` iterates standard TTYs (1-6) and serial consoles independently. If `/dev/tty0` exists as a standard device *and* `console=tty0` appears on the cmdline, it gets two inittab entries. A duplicated getty spawn on the same TTY will race for the device. Add a seen-set that tracks TTYs already written in the standard loop.

**Confidence: confirmed** — the Go code has no deduplication check between the standard loop and serial loop.

## Console I/O redirect absent

The original init script does:

```bash
if ! ${SCRIPTS}/bootstrap; then rescue; fi
exec >/dev/console </dev/console 2>&1
```

After bootstrap completes, stdin/stdout/stderr are redirected to `/dev/console`. The Go `Init.Run()` uses `slog` which writes to stderr, but never opens or redirects to `/dev/console`. On a real initramfs boot where the initial fd table points to `/dev/null` or a ramfs file, user-visible boot messages won't appear on the console.

**Confidence: confirmed** — no `/dev/console` open/redirect anywhere in the new code path.

## Side-effect coupling between mode detection and finalization

In `main.go` postChroot:

```go
ModeDetector: func() (string, error) {
    m, detectErr := detector.Detect(context.Background())
    if detectErr != nil {
        return "", detectErr
    }
    fin.Mode = m  // side effect
    return m, nil
},
```

The Finalizer's `Mode` field — which controls `GrowLive` behavior and `Cleanup` mode file writing — is set via a closure that captures `fin` by pointer. `Init` passes the mode string to `ModeRegistry` but never explicitly to `Finalizer`. If someone refactors `Init` to add a step between detect and finalize that early-returns, `fin.Mode` stays empty and `Cleanup` writes nothing. Consider having `Init` explicitly set a mode on the finalizer after successful detection, or pass mode as a parameter to `Finalizer.Run()`.

**Confidence: confirmed** — visible in `main.go` lines ~130-140.

## Disk Takeover control flow

The shell script's disk mode:

```bash
cd $TARGET
takeover || true    # may reboot/poweroff and never return
if [ -e k3os/system/factory-reset ] ... ; then
```

If `takeover` calls `reboot -f`, the process is replaced and execution stops. If it fails, `|| true` swallows the error and the script continues to ephemeral cleanup and pivot. The Go version returns an error from `Takeover()` that propagates up through `Execute()`:

```go
if err := h.Takeover(); err != nil {
    return fmt.Errorf("takeover: %w", err)
}
if err := h.CleanupEphemeral(); err != nil { ... }
if err := h.PivotAndExec(); err != nil { ... }
```

If `reboot -f` returns an error (container, dry-run), the entire disk mode aborts — unlike the shell which would continue. If somehow `Cmd.Run("reboot", "-f")` returns nil without replacing the process, execution continues into `CleanupEphemeral()` and `PivotAndExec()` with a half-deleted filesystem (factory-reset written, directories removed). The shell avoids this because either reboot works (process gone) or `|| true` keeps going without the deletion side-effects having occurred in a no-takeover path.

The fix: if the takeover marker doesn't exist, `Takeover()` returns nil early (correct). If it does exist and reboot/poweroff somehow fails, return a sentinel that `Execute()` can use to skip subsequent steps or re-attempt the reboot.

**Confidence: likely** — in production the process is dead after reboot, but the code structure permits the failure path.

## GrowPart error visibility

In `DiskHandler.SetupMounts`:

```go
_ = h.deps.Cmd.Run("parted", dev, "resizepart", num, "100%")
_ = h.deps.Cmd.Run("partprobe", dev)
_ = h.deps.Cmd.Run("e2fsck", "-f", devNum)
_ = h.deps.Cmd.Run("resize2fs", devNum)
```

All four grow commands discard errors. The shell also uses `grow ... || true` so failure is non-fatal, but there's zero logging here. A partition that fails to grow will silently run out of space later. At minimum these should `slog.Warn`.

In `finalize/grow.go`, `f.Cmd.Run("sleep", "2")` spawns a subprocess for a two-second delay. If `sleep` isn't in PATH (busybox not yet available, squashfs not mounted), the entire grow sequence errors out on what should be a trivial wait. Use `time.Sleep` or an injectable sleep func.

**Confidence: confirmed** — visible in disk.go lines 127-130 and grow.go line 55.
