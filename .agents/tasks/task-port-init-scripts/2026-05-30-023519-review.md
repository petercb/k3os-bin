# Shell-to-Go init port — v2 fix verification (v2)

All seven issues from the v1 review have been addressed in commit `445b223`. The fixes are structurally correct: `SetupRC` is in the right position, TTY deduplication uses a seen-set, `ConsoleRedirect` opens `/dev/console` and dup2's to fds 0/1/2, `ModeSetterFunc` makes mode propagation explicit, `Takeover` returns an error if reboot/poweroff doesn't replace the process, grow commands log via `slog.Warn`, and `SleepFunc` replaces the subprocess `sleep`.

Watch for: Finalizer dependency wiring gap — `CmdlineReader`, `RandFunc`, and `VirtDetector` are nil in production wiring (confirmed — will panic at runtime during `SetupTTYs`, `SetupHostname`, and `SetupServices`).

**Verdict**: NEEDS_CHANGES

## High-level view

The seven v1 fixes are correctly implemented and tested. One new issue emerged: the `Finalizer` struct in `postChroot()` is constructed without its three function-type dependencies (`CmdlineReader`, `RandFunc`, `VirtDetector`). These are called unconditionally by `SetupTTYs`, `SetupHostname`, and `SetupServices` respectively. A nil function call panics in Go — this is a guaranteed runtime crash on every boot.

## Finalizer wiring gap in postChroot

In `main.go` lines 117-122, the finalizer is constructed with only four fields set:

```go
fin := &finalize.Finalizer{
    FS:        fs,
    Mounter:   mounter,
    Cmd:       cmd,
    SleepFunc: time.Sleep,
}
```

The `Finalizer` struct declares three additional function fields used unconditionally by finalization steps:

- `CmdlineReader` — called by `SetupTTYs()` to parse `console=` entries
- `RandFunc` — called by `SetupHostname()` to generate a random hostname
- `VirtDetector` — called by `SetupServices()` to detect virtualization type

In Go, calling a nil `func` value panics immediately. The finalization step table runs `SetupTTYs` first, so the first panic will occur there on every production boot. Tests pass because each test constructs its own `Finalizer` with fields populated — the integration gap is purely in `postChroot` wiring.

The fix:

```go
fin := &finalize.Finalizer{
    FS:            fs,
    Mounter:       mounter,
    Cmd:           cmd,
    SleepFunc:     time.Sleep,
    CmdlineReader: readCmdline,
    RandFunc:      cryptoRandUint32,  // new helper
    VirtDetector:  detectVirt,        // new helper wrapping virt-what
}
```

**Confidence: confirmed** — `CmdlineReader` is visibly nil in `fin` and is called unconditionally by `SetupTTYs`.
