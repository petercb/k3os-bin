# Replace go-losetup with internal ioctl wrapper (v2)

The second pass addresses the three issues from v1: the data race on `Device.closed` is fixed via `sync/atomic.Bool`, `SetAutoclear` now short-circuits when the device is already detached, and `LOOP_CLR_FD` failure during `Attach` cleanup is logged.

**Watch for:** A TOCTOU window remains between `closed.Load()` and the subsequent ioctl in both `Detach` and `SetAutoclear` (possible тАФ narrow, needs concurrent timing). `Detach` itself is not atomic: a concurrent call can pass the `closed.Load()` guard and double-issue `LOOP_CLR_FD` (likely тАФ no compare-and-swap).

## High-level view

The v1 race concern is substantially mitigated. `atomic.Bool` eliminates the Go-level data race that would trigger the race detector, and the `closed` guard in `SetAutoclear` prevents ioctls on a known-closed fd. What remains is a narrow TOCTOU window: `Detach` can fire between `SetAutoclear`'s `closed.Load()` and its first ioctl. In this project's usage (single goroutine + single deferred detach), this is difficult to trigger тАФ the 3-second sleep makes it practically impossible for both to execute simultaneously. But `Detach` itself lacks a compare-and-swap, so two concurrent `Detach` calls would both pass the guard and both issue `LOOP_CLR_FD`.

## Detach atomicity gap

`Detach` reads `closed` and stores it in two separate operations:

```go
func (d *Device) Detach() error {
    if d.closed.Load() {   // T1 reads false
        return nil
    }
    // ... T2 also reads false here
    if err := d.sc.IoctlSetInt(d.fd, loopClrFd, 0); err != nil {
        return fmt.Errorf(...)
    }
    d.closed.Store(true)
    return d.sc.Close(d.fd)
}
```

If two goroutines call `Detach` concurrently, both pass the guard. The first succeeds; the second issues `LOOP_CLR_FD` on a fd that was just closed (or is about to be closed), getting `EBADF` or `ENXIO`. Then both call `d.sc.Close(d.fd)`, producing a double-close.

In the current `enterchroot` code, `Detach` is only called from the deferred closure тАФ never concurrently. But the package is exported via the `iface.LoopDevice` interface, and nothing prevents future callers from triggering this. The idiomatic fix is `if !d.closed.CompareAndSwap(false, true) { return nil }` at the top, making the first caller win atomically and eliminating the close-then-ioctl and double-close paths.

## SetAutoclear TOCTOU тАФ acceptable for current usage

The window between `closed.Load()` returning false and the `IoctlLoopGetStatus64` call is real but extremely narrow. In the `enterchroot` goroutine, `SetAutoclear` fires 3 seconds after `Attach`. `Detach` only fires when `Mount` returns (either via error or after `cmd.Run()` completes). For the race to hit, `Mount` must return within microseconds of the 3-second mark тАФ possible but unlikely in practice. The consequence is a logged `EBADF` error, not data corruption or a stuck device.
