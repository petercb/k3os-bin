# Replace go-losetup with internal ioctl wrapper

The external `github.com/freddierice/go-losetup/v2` dependency is replaced by a focused internal package (`internal/loopdev`) that wraps the five Linux loop device ioctls directly via `golang.org/x/sys/unix`. The old multi-step `GetInfo`/`SetInfo`/flag-manipulation pattern in `enterchroot` collapses into a single `SetAutoclear()` call. The package is designed around a `syscaller` interface seam that makes unit testing possible without root privileges or real loop devices.

**Watch for:** Race between the `SetAutoclear` goroutine and the deferred `Detach` call ‚Äî if `Mount` returns quickly (error or fast exec), `Detach` closes the fd while `SetAutoclear` is still sleeping (confirmed). `SetAutoclear` operating on a closed/detached device is undefined (confirmed ‚Äî no `closed` guard). `loopInfo64` struct layout verified correct against kernel headers.

## High-level view

The `enterchroot` integration is a surgical replacement ‚Äî `losetup.Attach` becomes `loopAttacher.Attach`, and the `GetInfo`/`SetInfo`/flag-toggle sequence becomes `dev.SetAutoclear()`. The new code logs `SetAutoclear` failures (previously silently swallowed), but the goroutine that calls `SetAutoclear` after a 3-second sleep races with the deferred `Detach` in error paths.

Resource management in `Attach` is careful on the happy path but has a gap: if `LOOP_SET_STATUS64` fails and the subsequent cleanup `LOOP_CLR_FD` also fails, the kernel-side loop device association persists with no userspace handle to clean it up. The error is discarded silently.

The test suite covers all error injection points for `Attach`, `Detach`, and `SetAutoclear`, but does not exercise the `SetAutoclear`/`Detach` race or the `LOOP_SET_STATUS64` failure cleanup path.

## SetAutoclear/Detach race in enterchroot

In `enter.go`, `Mount` attaches the loop device, defers `dev.Detach()`, then spawns a goroutine that sleeps 3 seconds before calling `dev.SetAutoclear()`. If `Mount` returns before those 3 seconds elapse (error during reexec setup, or when PID == 1 and `syscall.Exec` is reached immediately), the deferred `Detach` fires first. `Detach` sets `d.closed = true` and calls `Close(d.fd)`. Three seconds later the goroutine wakes and calls `SetAutoclear`, which issues `IoctlLoopGetStatus64` on a closed fd.

The pre-existing code had the same temporal structure (sleep 3s, then set flags), but the old code silently swallowed errors. The new code logs the error, which is better, but the underlying race now surfaces as a `EBADF` log line on fast-exit paths. The fix is straightforward: either check `d.closed` at the top of `SetAutoclear`, or use a context/channel to cancel the goroutine when `Detach` is called.

## Device lifecycle ‚Äî missing closed guard on SetAutoclear

`Detach` checks `d.closed` before operating. `SetAutoclear` does not. This asymmetry means any caller of `SetAutoclear` on a previously-detached device will issue raw ioctls on a closed (or recycled) file descriptor. In the current codebase only the goroutine in `enterchroot` hits this, but the interface contract doesn't prevent future callers from triggering it. Adding `if d.closed { return fmt.Errorf("device already detached") }` at the top of `SetAutoclear` makes the behavior consistent with `Detach`.

Note: `d.closed` is not protected by a mutex. Since the goroutine and the deferred `Detach` run concurrently, reading `d.closed` without synchronization is itself a data race. A minimal fix would be `sync.Once` for the close path or an `atomic.Bool` for the flag.

## Attach cleanup on LOOP_SET_STATUS64 failure

When `LOOP_SET_STATUS64` fails after `LOOP_SET_FD` succeeded:

```go
_ = a.sc.IoctlSetInt(loopFd, loopClrFd, 0)
_ = a.sc.Close(loopFd)
```

Both errors are discarded. If `LOOP_CLR_FD` fails (kernel returns EBUSY during a race with udev probing the device), the loop device remains associated in the kernel's loop table. The fd is closed, so userspace loses the handle. This is a narrow edge case at boot time, but logging the `LOOP_CLR_FD` error would help debug stuck loop devices.

## Test coverage gaps

Not tested: the `SetAutoclear`-after-`Detach` race (would require injecting a controllable clock or a synchronization point). Not tested: `SetAutoclear` called on a closed device. Not tested: `LOOP_SET_STATUS64` failure path where `LOOP_CLR_FD` also fails.
